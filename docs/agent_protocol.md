# Agent Handshake Protocol

Короткий контракт общения между агентами «Маркетолог» и «Контент-мейкер». Используй этот файл как источник истины при настройке связи агентов, сценариев B/D и передачи результата одного агента другому.

## Цель
- Не тратить токены на повторное объяснение ролей и формата обмена.
- Передавать только результат, ограничения и открытые вопросы.
- Не передавать внутренние рассуждения, черновики и историю диалога без прямой необходимости.

## Участники
- `marketer` — отвечает за стратегию: ЦА, УТП, позиционирование, KPI, каналы, ограничения.
- `content_maker` — превращает стратегию в материалы: посты, письма, тексты, сценарии, промпты.

## Сообщение
Каждая передача между агентами должна быть коротким JSON-объектом. Для `strategy_to_content` это строгий контракт: без Markdown-обёртки, без пояснений до/после JSON, без chain-of-thought.

```json
{
  "from": "marketer",
  "to": "content_maker",
  "task_id": "string",
  "project_id": "string",
  "stage": "strategy_to_content",
  "summary": "1-3 предложения: что нужно сделать дальше",
  "inputs": {
    "audience": "для кого",
    "offer": "что продаём или продвигаем",
    "positioning": "ключевой угол",
    "constraints": ["тон", "канал", "формат", "объём", "дедлайн"],
    "must_include": ["обязательные смыслы"],
    "must_avoid": ["запрещённые смыслы"]
  },
  "expected_output": {
    "format": "markdown | json | html | plain_text",
    "deliverables": ["что должен вернуть следующий агент"],
    "acceptance_criteria": ["как понять, что результат подходит"]
  },
  "open_questions": []
}
```

## Правила
- `marketer -> content_maker` всегда передаётся как валидный JSON, который можно распарсить через `JSON.parse`.
- Если `content_maker` не может распарсить JSON handoff, workflow должен завершиться ошибкой вместо попытки восстановить смысл из markdown.
- Агент-получатель доверяет полям `summary`, `inputs` и `expected_output` как контракту задачи.
- Если данных достаточно, агент выполняет задачу без повторного handshake.
- Если данных недостаточно, агент возвращает только `open_questions`, без попытки угадать недостающий контекст.
- Для сценария B используется один переход: `marketer -> content_maker`.
- Для сценария D каждый цикл передаёт только дельту: что принято, что исправить, какие ограничения изменились.
- В сообщении нельзя передавать chain-of-thought, приватные заметки, полные логи и внутренние детали реализации агентов.

## Стадии
- `strategy_to_content` — Маркетолог передаёт стратегию Контент-мейкеру.
- `content_to_review` — Контент-мейкер передаёт результат на проверку Маркетологу или Evaluator.
- `review_to_revision` — проверяющий агент передаёт точечные правки.
- `final_delivery` — агент возвращает финальный результат оркестратору.
